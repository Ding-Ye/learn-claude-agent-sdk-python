# s03 — Parser

## Problem

transport 给我们的是字符串。每行字符串应该是一个 JSON 对象代表一条 message——但内容是多态的（一个 `assistant` 消息可以同时包含 `text` 块、`tool_use` 块、`thinking` 块）。要把它落到类型化的 Go 值上，又不想写一个 3 页的 switch。

## Solution

`Parse(line []byte) (Message, error)`。两步：先用一个小匿名 struct peek `type` 字段，再分发到对应解码器。content block 也走同样的模式。遇到不认识的 content block 静默丢弃（跟上游的前向兼容策略一致）。

## How It Works

`json.RawMessage` 是让两步法成立的关键：peek 时把 bytes 留着不解析，分发后再按正确 struct 解一次。对 content block，先 peek 它的 `type`，不认识就 skip——这样新 CLI 版本不会把老 SDK 撞挂。

## What Changed

- 新增模块 `agents/s03-parser`。
- `parser.go` 内联了 message + block 类型定义（不跨章 import），给出 `Parse` 和 `ParseError`。
- `parser_test.go` 覆盖四种 message 类型 + 未知 block 跳过的语义。

## Try It

```
cd agents/s03-parser
go test ./...
```

七个测试通过，其中一个故意塞一个未知 block，验证它被跳过、不挂。
