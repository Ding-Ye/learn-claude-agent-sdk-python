# s06 — Tools & Permissions

## Problem

The CLI is about to call a tool. Should it? "Yes" can mean: pre-approved by an allowlist, or by mode, or by a user-supplied callback. "No" can mean: blocklisted, or default-deny. We need a deterministic, debuggable decision chain.

## Solution

A `Policy{Allowed, Disallowed, Mode, CanUseTool}` and a single `Evaluate(ctx, name, input)` method. Six-step chain: deny-list → allow-list → bypass → dontAsk → callback → default-deny.

## How It Works

Order matters. Disallow wins over allow so a guardrails team can blocklist a tool without worrying about who allow-listed it. The callback is the **last** programmatic gate — if neither list nor mode decides, the user gets the call.

## What Changed

- New module `agents/s06-tools-permissions`.
- `perm.go`: `Policy`, `Decision`, `CanUseTool`, `Policy.Evaluate`.
- Nine tests for every branch in the decision chain.

## Try It

```
cd agents/s06-tools-permissions
go test ./...
```

The tests assert `disallow` wins over `allow`, `bypass`/`dontAsk` work, the callback runs only when needed, callback errors bubble, and wildcards (`"*"`) match anything.
